  ---
  cc/core/ 核心架构深度解析

  一、整体定位：三层架构

  main.py (控制面：REPL循环、用户交互)
      ↓
  QueryEngine (枢纽层：状态容器、依赖装配)
      ↓
  query_loop (状态机内核：while(true) 循环)
      ↓
  stream_response (API适配层：SSE事件 → QueryEvent)
      ↓
  StreamingToolExecutor (工具编排层：并发控制、hooks、权限)

  为什么是三层而不是两层？ 如果 query_loop
  直接持有状态（client、messages、model），它就无法被复用了——子 agent、测试 mock
   都需要不同的配置。把状态外推到 QueryEngine，让 query_loop
  保持纯函数签名（所有依赖通过参数注入），是这个架构最核心的设计决策。

  ---
  二、核心数据结构及其设计理由

  2.1 消息模型 (models/messages.py)

  Message = UserMessage | AssistantMessage | SystemMessage |
  CompactBoundaryMessage

  四种消息，为什么不是一种？

  ┌────────────────┬────────┬───────────┬──────────────────────────────────┐
  │      类型      │ 角色角 │ 发给API？ │             存在理由             │
  │                │   色   │           │                                  │
  ├────────────────┼────────┼───────────┼──────────────────────────────────┤
  │ UserMessage    │ user   │ 是        │ 用户输入 + tool_result 回传（API │
  │                │        │           │ 要求tool_result在user角色中）    │
  ├────────────────┼────────┼───────────┼──────────────────────────────────┤
  │ AssistantMessa │ assist │ 是        │ 模型输出（文本 + tool_use        │
  │ ge             │ ant    │           │ 调用请求）                       │
  ├────────────────┼────────┼───────────┼──────────────────────────────────┤
  │ SystemMessage  │ 无     │ 否        │ 本地通知（权限提示、hook反馈）， │
  │                │        │           │ 不需要让模型看到                 │
  ├────────────────┼────────┼───────────┼──────────────────────────────────┤
  │ CompactBoundar │ 无     │ 转换为use │ 压缩边界标记，在发给API时被转换  │
  │ yMessage       │        │ r消息     │ 为带摘要的user消息               │
  └────────────────┴────────┴───────────┴──────────────────────────────────┘

  设计理由： 这四种消息对应了对话系统中完全不同的生命周期。SystemMessage
  从不离开本地（normalize_messages_for_api 过滤掉），CompactBoundaryMessage
  在转换边界处改变形态。如果用一种通用消息 + flag 来区分，所有消费方都需要大量的
   if/else 判断。用类型系统来表达语义，是让编译器（mypy）帮你检查正确性。

  2.2 内容块 (models/content_blocks.py)

  ContentBlock = TextBlock | ToolUseBlock | ToolResultBlock | ThinkingBlock |
  RedactedThinkingBlock | ImageBlock

  为什么要 7 种内容块而不是通用一个？ 直接映射 Anthropic API 的 content block
  格式。每种块有完全不同的字段（ToolUseBlock 有 id/name/input，ImageBlock 有
  source），硬塞进一个通用结构只会增加解析复杂度。

  关键约束：

  AssistantContentBlock = TextBlock | ToolUseBlock | ThinkingBlock |
  RedactedThinkingBlock
                        ↑ 不包含 ToolResultBlock 和 ImageBlock

  UserContentBlock = TextBlock | ToolResultBlock | ImageBlock
                   ↑ 不包含 ToolUseBlock

  为什么要有这个约束？ 因为 Anthropic API 要求：
  - tool_result 只能出现在 user 角色消息中
  - tool_use 只能由模型生成（assistant 角色）
  - 类型系统强制执行这个约束，在消息构造阶段就能拦截错误

  每个块都有 to_api_dict() / from_api_dict() 对——为什么？ 因为内部表示和 API
  表示之间存在映射关系。例如 ToolResultContent 的 from_api_dict
  对未知类型做了降级处理（content_blocks.py:88-94），这种防御性逻辑放在数据层而
  非业务层，保证了序列化/反序列化的完备性。

  2.3 事件系统 (core/events.py)

  QueryEvent = TextDelta | ThinkingDelta | ToolUseStart | ToolResultReady |
  CompactOccurred | TurnComplete | ErrorEvent

  事件系统存在的核心理由：解耦。

  query_loop 是一个 async generator，通过 yield 产出事件。它 永远不直接写 
  stdout。UI 层（ui/renderer.py）消费这些事件来渲染终端输出。这意味着：
  - 换一个 UI（Web、IDE插件）不需要改 query_loop 一行代码
  - 测试时可以收集事件序列来验证行为，不需要 mock stdout

  为什么用 Union 类型而不是继承体系？ 代码注释 (events.py:14-17)
  解释了三个原因：
  1. 事件种类有限且封闭（7种，不需要外部扩展）
  2. isinstance 检查比 visitor 模式更 Pythonic
  3. mypy 能对 Union
  做穷举检查——如果消费方漏处理了某个事件类型，类型检查器会报错

  事件的生命周期顺序：

  一次完整turn:
    TextDelta/ThinkingDelta → ToolUseStart → TurnComplete → ToolResultReady →
  (下一轮turn)

  特殊情况:
    CompactOccurred 在 API 调用前触发（Phase 1）
    ErrorEvent 可在任意阶段触发（Phase 2 流式过程中，或 Phase 3 集中处理）

  2.4 工具系统 (tools/base.py)

  Tool (抽象基类)
  ├── get_name() → str          # API层面的唯一标识
  ├── get_schema() → ToolSchema # 告诉模型这个工具能做什么、需要什么参数
  ├── execute(input) → ToolResult  # 实际执行
  └── is_concurrency_safe(input) → bool  # 能否和其他工具并行

  ToolRegistry
  ├── register(tool)            # O(1) 注册
  ├── get(name) → Tool | None   # O(1) 查找
  └── get_api_schemas() → list  # 批量导出给API

  ToolResult 为什么不抛异常而是返回 is_error=True？ 这是整个 agent
  系统的关键设计。如果工具执行失败抛异常，query_loop 的整个循环就会中断。但
  agent 的核心能力之一就是从错误中恢复——模型看到 is_error=True
  的结果后，可以决定重试、换参数、或向用户解释。异常应该只用于"系统级不可恢复的
  故障"（如网络断开），而不用于"工具执行的业务错误"（如文件不存在）。

  ToolRegistry 为什么存在？ 解耦工具发现和工具使用。query_loop
  不持有任何具体工具实例，只通过 registry.get(name) 动态查找。这让同一个
  query_loop 代码可以服务于不同的工具集——主 agent 有全部22个工具，子 agent 排除
  AgentTool（防递归），后台 agent 排除 AskUser（非交互）。

  2.5 应用状态 (models/state.py)

  AppConfig         # 全局配置（API key、模型、路径、会话ID）
  QueryState        # 每轮循环的不可变快照
  AutoCompactTracking # 压缩追踪（连续失败次数、上次压缩轮次）
  ThinkingConfig    # 扩展思考配置

  为什么 QueryState 是不可变的？ 对应 TypeScript 原版中每轮循环创建新的 State
  对象的模式。虽然 Python 版 query_loop 实际上用的是局部变量而非 QueryState
  实例（简化实现），但数据结构保留了。不可变状态的最大好处是：你可以安全地检查"
  这一轮开始时的状态是什么"，不用担心中途被修改。

  ---
  三、query_loop 状态机——逐阶段分析

  while turn_count < max_turns:
      turn_count += 1

      Phase 1: 准备（消息规范化 + auto-compact）
      Phase 2: 调用模型（流式API + 工具提前执行）
      Phase 3: 错误恢复
      Phase 4: 工具结果拼回

      → continue 或 return

  Phase 1：准备

  api_messages = normalize_messages_for_api(messages)

  为什么需要 normalize？ 因为内部 messages 列表是"自由格式"的——可能包含
  SystemMessage（需要过滤）、连续的 user 消息（需要合并）、缺失配对的
  tool_use/tool_result（需要修补）。Anthropic API 要求三条硬约束：
  1. 消息必须 user/assistant 交替
  2. 对话必须以 user 开头
  3. 每个 tool_use 必须有对应的 tool_result

  normalize 的三步修复逻辑 (messages.py:184-230)：
  - Pass 1：角色交替修复（连续 user 合并，连续 assistant 插占位符）
  - Pass 2：确保以 user 开头
  - Pass 3：tool_use/tool_result 配对修复（三步双向：收集ID → 移除孤儿result →
  为孤儿use补充合成result）

  auto-compact 的触发时机为什么在 Phase 1？ 因为在调 API 之前检查最合理——如果
  token 量已经接近上限（70% 阈值），先压缩再调用，避免触发 API 的 413
  错误（prompt_too_long），后者是被动恢复，成本更高。

  Phase 2：调用模型（流式）

  async for event in call_model(messages=api_messages, system=system_prompt,
  tools=tool_schemas, max_tokens=current_max_tokens):

  为什么用 async generator 而不是返回完整结果？
  流式响应的核心理由是延迟优化。模型在输出第一个 token 时就开始 yield
  TextDelta，UI 立即显示。更重要的是
  ToolUseStart——当一个工具调用块完整接收时（content_block_stop），query_loop
  立即启动该工具的执行，不等整个 API 响应完成。如果模型返回 3 个工具调用，第 1
  个可能在第 3 个还在传输时就已经执行完了。

  call_model 为什么是闭包而不是直接传 client？ 三个理由：
  1. query_loop 只需要一个"能调用模型"的能力，不关心具体实现
  2. 子 agent 可以用不同的 model 参数创建不同的 call_model
  3. auto_compact 需要一个 max_tokens=4096 的低配版（压缩摘要不需要长输出，省
  token）

  Phase 3：错误恢复

  设计原则：可恢复错误不消耗轮次预算。

  if recovered:
      turn_count -= 1  # 回退轮次计数，"这轮不算"
      continue

  为什么不直接 retry_count++ 然后 continue？ 因为如果用 retry_count
  来控制退出，那 5 次重试就会消耗 5 轮 max_turns 预算。在极端情况下（频繁
  429），用户设置的 max_turns=10 可能全部被重试耗尽，一次有效执行都没有。分离
  turn_count 和 retry_count 是为了保证"有效执行次数"不被错误恢复偷走。

  四种恢复策略的设计理由：

  ┌────────────┬────────┬──────────────────────────────────────────────────┐
  │  错误类型  │ 恢复方 │                  为什么这样恢复                  │
  │            │   式   │                                                  │
  ├────────────┼────────┼──────────────────────────────────────────────────┤
  │ prompt_too │ 响应式 │ Phase 1 的主动 compact                           │
  │ _long      │  compa │ 没拦住，说明阈值判断有偏差，强制压缩一次。只允许 │
  │ (413)      │ ct     │ 一次，防压缩-重试-压缩死循环                     │
  ├────────────┼────────┼──────────────────────────────────────────────────┤
  │ max_output │ escala │ 第一次截断可能是 max_tokens 设太低，提升限制可能 │
  │ _tokens    │ te 16K │ 就够了，不需要追加"请继续"消息                   │
  │ (首次)     │ →64K   │                                                  │
  ├────────────┼────────┼──────────────────────────────────────────────────┤
  │ max_output │ 保存已 │                                                  │
  │ _tokens    │ 输出 + │ 已经 64K 还截断，说明内容确实很长。保存已有输出  │
  │ (后续)     │  "请继 │ 让模型接续，避免从头生成                         │
  │            │ 续"    │                                                  │
  ├────────────┼────────┼──────────────────────────────────────────────────┤
  │            │ 指数退 │                                                  │
  │ 429/529    │ 避     │ 瞬时错误，等一等就好。线性增长而非指数，是因为   │
  │            │ (2s×re │ API 限流窗口通常不长，2-10s 足够                 │
  │            │ try)   │                                                  │
  └────────────┴────────┴──────────────────────────────────────────────────┘

  Phase 4：工具执行与结果拼回

  if tool_use_blocks:
      tool_results = await executor.get_results()
      # ... 构建 ToolResultBlock ...
      messages.append(UserMessage(content=list(result_blocks)))  # ← 作为 user 
  消息
      continue  # 带着结果继续下一轮

  三个退出条件和三个 continue：

  退出 1: error_event 不可恢复 → yield ErrorEvent + return
  退出 2: 无工具调用（stop_reason != tool_use）→ return（正常结束）
  退出 3: turn_count >= max_turns → yield ErrorEvent（超限）

  continue 1: Phase 3 恢复成功 → 重试本轮
  continue 2: max_tokens 正常截断 → 追加"请继续"
  continue 3: Phase 4 有工具调用 → 拼回结果后继续

  ---
  main.py 的角色：装配工 + 司机

  main.py 自己不实现任何对话逻辑。它做两件事：

  1. 装配（启动时）：把散落的组件（API
  client、工具、prompt、权限、hooks）焊成一台能跑的机器（QueryEngine）
  2. 驱动（运行时）：循环读取输入 → 交给引擎 → 消费事件 → 渲染 UI

  ---
  启动流程（装配阶段）

  main()
    ↓

    ↓

  两种运行模式

  Print 模式（-p）：最简路径

  async def _run_print_mode(prompt, model):
      engine = _build_engine(client, model, cwd, is_interactive=False)
      async for event in engine.submit(prompt):
          render_event(event)

  submit() 内部自动把 prompt 包装成 UserMessage，跑一轮

  command。这就是为什么它标注 is_interactive=False——权限全部自动放行。

  REPL 模式（默认）：完整路径

  主循环结构，每轮做 5 件事：

  while True:
      A. 读输入（支持多行）
      B. 分派处理：
         ├─ /command → 直接处理（clear/compact/model/skill），不走 query_loop
         └─ 普通文本 → append UserMessage 到 transcript
      C. 收件箱轮询（仅 team 模式，拉取 teammate 消息）
      D. engine.run_turn() → 驱动 query_loop，消费事件流
      E. 后处理：
         ├─ save_session()：持久化 transcript
         └─ asyncio.create_task(_bg_extract())：后台提取 memory

  自己的上下文，它做不到。这些操作必须在 REPL 层直接执行。

❯ 什么叫二次布线，又为什么要使用call_model_factory                              

● 二次布线

  问题出在依赖顺序。看 _build_registry 和 _build_engine 的执行顺序：

  _build_engine() {
      Step 4: task_registry = TaskRegistry()       ← 这时候才创建
      Step 4: bg_manager = BackgroundAgentManager(task_registry)
      Step 5: registry = _build_registry(...)       ← 在这里注册工具
      Step 7: AgentTool 注入 bg_manager             ← 这时候才能注入
  }

  AgentTool 的构造需要 bg_manager，但 bg_manager 是在 _build_engine 的 Step 4
  创建的，而 _build_registry 在 Step 5 才被调用。

  所以 _build_registry 注册 AgentTool 时，bg_manager
  还不存在。只能先注册一个缺依赖的"骨架版"，等 Step 4 创建完 bg_manager 后，在
  Step 7 用完整版替换：

  # Step 5: 骨架版（缺 bg_manager、cwd）
  registry.register(AgentTool(parent_registry=registry, call_model_factory=...,
  model=...))

  # Step 7: 完整版（有 bg_manager、cwd）
  registry._tools.pop("Agent")          # 删掉骨架版
  registry.register(AgentTool(           # 注册完整版
      parent_registry=registry,
      call_model_factory=...,
      model=...,
      bg_manager=bg_manager,            # ← 这时候有了
      cwd=cwd,                          # ← 这时候有了
  ))

  为什么不让 _build_registry 接收 bg_manager 参数？ 因为 bg_manager 本身又依赖
  TaskRegistry，而 TaskRegistry 的创建逻辑属于 _build_engine
  的职责。如果你把这些都塞进 _build_registry，它就会变成一个巨大的上帝函数。二次
  布线是用一小段重复代码换取模块边界的清晰。

  ---
  call_model_factory

  先看 query_loop 怎么调用模型：

  async for event in call_model(messages=..., system=..., tools=...,
  max_tokens=...):

  call_model 是一个函数，签名是 (**kwargs) -> AsyncIterator[QueryEvent]。

  最简单的方式：直接在 query_loop 里写 client.messages.stream(model=..., ...)。

  但这样 query_loop 就绑死了 Anthropic SDK。测试时你得 mock 整个 SDK。子 agent
  想用不同模型怎么办？compact 想用低配版（max_tokens=4096）怎么办？

  所以用闭包把 client 和 model 包起来：

  def make_call_model(model, max_tokens=16384):
      client = self._client  # 闭包捕获

      async def call_model(**kwargs):
          effective_max = kwargs.pop("max_tokens", max_tokens)
          async for event in stream_response(client, model=model,
  max_tokens=effective_max, **kwargs):
              yield event

      return call_model

  query_loop 只拿到一个"能产出事件的函数"，不知道也不关心底层是谁。

  但 AgentTool 还有一层需求：子 agent
  可能在运行时才知道要用什么模型。make_call_model 需要在调用时才知道 model
  参数，而不是在创建时就固定。

  所以再包一层——call_model_factory：

  # 第2层：绑定 model
  call_model = engine.make_call_model(model="claude-sonnet-4-20250514")
  # 这个 call_model 的 model 已经固定了

  # 第3层：延迟绑定 model
  factory = engine.make_call_model_factory()
  # factory 本身不绑定 model，等调用时再决定：
  call_model_for_sub_agent = factory(model="claude-haiku-4-5-20251001")
  call_model_for_compact = factory(max_tokens=4096)

  实际场景：AgentTool 在执行时读取用户指令里的 model 字段（比如
  sonnet、haiku），然后调 factory(model="claude-haiku-4-5-20251001") 创建子
  agent 专用的 call_model。如果没有 factory，AgentTool 就无法动态选择模型。

  ---
  总结：二次布线解决的是"注册时依赖还没创建"的时序问题；call_model_factory解决的
  是"运行时才确定用哪个模型"的延迟绑定问题。两者都是用一层间接换取灵活性。
