# 第二步：入口与装配（main.py）

## main.py 的角色：装配工 + 司机

`main.py` 自己不实现任何对话逻辑。它做两件事：

1. **装配**（启动时）：把散落的组件（API client、工具、prompt、权限、hooks）焊成一台能跑的机器（`QueryEngine`）
2. **驱动**（运行时）：循环读取输入 → 交给引擎 → 消费事件 → 渲染 UI

---

## 启动流程

```
__main__.py（7行）
  → from cc.main import main → main()

main()
  ├─ _load_env() → 读 .env + 环境变量，拿到 API key
  ├─ create_client(api_key) → 创建 Anthropic SDK 客户端
  ├─ _build_engine(client, model, cwd) ← 装配的核心
  └─ _connect_mcp_servers() → 连接外部 MCP 工具服务器
```

## _build_engine：8步装配

这是整个系统最关键的装配函数，把散落组件焊成一台可以运转的机器。

### 为什么需要 _build_engine？

`query_loop` 是纯函数（所有依赖通过参数注入），不持有任何状态。
但一次完整对话需要的状态（messages、client、registry 等）必须有地方管理。
`_build_engine` 就是做这个"装配"的——它把所有组件准备好，打包成一个 `QueryEngine` 实例。

### 8步装配顺序

```
Step 1: 基础构建
  ├─ load_claude_md(cwd)           → 从 cwd 向上搜索 CLAUDE.md
  ├─ build_system_prompt(...)      → 拼装完整 system prompt
  ├─ maybe_inject_coordinator_prompt() → 如果处于 coordinator 模式，注入编排指令
  └─ load_hooks()                  → 从 ~/.claude/settings.json 读取 hook 配置

Step 2: 权限上下文
  └─ PermissionContext(mode=ACCEPT_EDITS, is_interactive=...)
     ├─ ACCEPT_EDITS: 文件编辑自动放行，危险操作仍需确认
     └─ is_interactive=False 时：所有权限请求自动放行（print mode / 子 agent）

Step 3: 运行时基础设施
  ├─ TaskRegistry         → 跟踪所有后台任务的状态机
  ├─ BackgroundAgentManager → 管理后台 agent 的生命周期
  └─ TeamContext           → 可变团队状态容器（team_name、成员列表）
     → 创建时是空的，TeamCreateTool 触发时才激活

Step 4: call_model_factory（解决鸡生蛋问题）
  ├─ engine_ref = []  （用 list 当可变容器）
  ├─ factory 闭包捕获 engine_ref
  └─ engine 创建后执行 engine_ref.append(engine)
     → factory 从此能访问 engine，走 engine.make_call_model()

Step 5: 工具注册（第一遍）
  └─ _build_registry(cwd, call_model_factory, model)
     → 注册所有 22 个工具的"骨架版"

Step 6: 二次布线
  ├─ AgentTool:   pop 骨架 → register 完整版（注入 bg_manager + cwd）
  ├─ TeamCreate:  注入 team_context
  ├─ TeamDelete:  注入 team_context
  ├─ SendMessage: 注入 team_context
  └─ TaskStop:    注入 task_registry

Step 7: 组装 QueryEngine
  └─ QueryEngine(client, model, registry, system_prompt, hooks, permission_ctx)
     + engine_ref.append(engine) ← 让 factory 闭包从此能访问 engine
```

### 为什么需要"二次布线"？

因为存在循环依赖：

```
_build_registry() 需要 call_model_factory
call_model_factory 需要 engine（通过 engine_ref）
engine 需要 registry（构造参数）
```

解决办法是分两步：
1. 先注册骨架版工具（缺少运行时依赖）
2. engine 创建后，用完整版工具 pop + register 覆盖

代码体现：`registry._tools.pop(name)` + `registry.register(完整版工具)`

### engine_ref 闭包技巧

```python
engine_ref: list[QueryEngine] = []

def _factory(m, max_tokens=16384):
    if engine_ref:                                    # engine 已创建
        return engine_ref[0].make_call_model(...)
    return _make_call_model(client, m or model, ...)   # 降级路径

# ... engine 创建 ...
engine = QueryEngine(...)
engine_ref.append(engine)  # 此时 factory 闭包才能访问 engine
```

用 list 而不是直接用变量，是因为 Python 闭包捕获的是变量引用而非值。
`engine_ref` 列表在闭包创建前就存在（虽然是空的），闭包可以安全地读取它。
而 `engine` 变量在闭包创建时还不存在，直接捕获会报 UnboundLocalError。

## call_model 三层抽象

这是理解整个系统依赖注入的关键：

```
第1层: stream_response(client, model, max_tokens, messages, ...)
       ↑ 最底层，直接调 Anthropic SDK，返回 AsyncIterator[QueryEvent]

第2层: call_model(**kwargs) → AsyncIterator[QueryEvent]
       ↑ 闭包，绑定了 client + model + max_tokens
       ↑ query_loop 只认识这个签名，不关心底层是什么

第3层: call_model_factory(model, max_tokens) → call_model
       ↑ 工厂的工厂，给 AgentTool 用来给子 agent 创建不同配置的 call_model
```

**为什么 query_loop 不直接拿 client？**
1. 解耦：测试时可以注入 mock_call_model，不需要真实 API 调用
2. 子 agent 复用同一个 factory，自动继承父 agent 的 client 配置
3. auto_compact 需要一个 max_tokens=4096 的低配版（压缩摘要不需要长输出）

## _build_registry：工具分层注册

### 三层工具

```
Tier 1 — 核心文件操作（顶层 import，启动就绑定）
  BashTool, FileReadTool, FileEditTool, FileWriteTool, GlobTool, GrepTool

Tier 2 — 扩展工具（lazy import，减少启动时间和循环依赖）
  TaskCreate/Get/List/Update/Stop, WebFetch, AskUser, TodoWrite,
  WebSearch, NotebookEdit, ToolSearch, SkillTool, PlanMode, Brief, LSP

Tier 3 — 协作工具（需要 call_model_factory）
  AgentTool, TeamCreate, TeamDelete, SendMessage
```

### 为什么 Tier 2 要 lazy import？

1. 避免不必要的依赖：`notebook` 依赖 `nbformat`，用户可能没装——延迟导入让不用的功能不报错
2. 避免循环依赖：某些工具 import 了 `main.py` 的组件

### ToolSearchTool 为什么传入 registry？

因为它的功能是"搜索可用工具"——它需要能读取当前 registry 的内容。
这是唯一一个依赖 registry 本身的工具。

## 两种运行模式

### Print 模式（`-p`）：最简路径

```python
async def _run_print_mode(prompt, model):
    engine = _build_engine(client, model, cwd, is_interactive=False)
    async for event in engine.submit(prompt):
        render_event(event)
```

- `submit()` 内部自动把 prompt 包装成 `UserMessage`
- 跑一轮 `query_loop`，完事退出
- 没有 session 持久化、没有 memory extraction、没有 slash command
- `is_interactive=False` → 权限全部自动放行

### REPL 模式（默认）：完整路径

```python
while True:
    A. 读输入（支持多行：行尾\、括号不平衡、三引号未闭合）
    B. 分派处理：
       ├─ /command → 直接在 REPL 层处理（clear/compact/model/skill）
       └─ 普通文本 → messages.append(UserMessage(content=user_input))
    C. 收件箱轮询（仅 team 模式，拉取 teammate 消息注入 transcript）
    D. engine.run_turn() → 驱动 query_loop，消费事件流渲染 UI
    E. 后处理：
       ├─ save_session()：持久化 transcript + task 状态
       └─ asyncio.create_task(_bg_extract())：后台提取 memory（不阻塞下一轮输入）
```

### 为什么 slash command 不走 query_loop？

因为 `/clear`、`/model` 这类命令是**改变运行时状态**的，不是对话内容。
如果发给模型，模型会困惑——你让它清空自己的上下文，它做不到。
这些操作必须在 REPL 层直接执行。

每个 command handler 返回一个标记字符串：
- `__CLEAR__` → messages.clear()
- `__COMPACT__` → 手动触发 compact
- `__MODEL__xxx` → 切换模型 + 重建 system prompt
- `__SKILL__xxx` → 把 skill prompt 注入 transcript

### 后处理：后台 memory extraction

```python
task = asyncio.create_task(_bg_extract())
_bg_tasks.add(task)                   # 持有引用防止 GC
task.add_done_callback(_bg_tasks.discard)  # 完成后自动移除
```

- 用低配 call_model（max_tokens=1024）扫描最近对话，提取值得长期记忆的信息
- 完全是异步的，不阻塞下一轮用户输入
- `_bg_tasks` set 持有 task 引用，防止 Python 的 GC 回收未完成的协程

## _build_system：system prompt 的组成

`build_system_prompt()` 返回一个 `list[str]`，每个元素是一个段落，按顺序拼接：

```
1. Intro      — 身份声明 + 安全指令
2. System     — 工具选择、输出格式等系统级行为
3. Doing tasks — 任务执行原则
4. Actions    — 高风险操作的确认规范
5. Using tools — 工具使用规范（优先用专用工具而非 Bash）
6. Tone/Style — 语气和风格
7. Output efficiency — 简洁输出
8. Environment — 运行时环境信息（cwd、OS、model 等）
9. SUMMARIZE   — 工具结果摘要提示
10. Memory      — memory 行为指导 + MEMORY.md 索引内容
11. CLAUDE.md   — 用户自定义指令
```

最终用 `\n\n` 拼成一个字符串，作为 API 请求的 `system` 参数。
